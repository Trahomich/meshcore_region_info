# region home — подробное руководство

## Что это

`region home <регион>` — это объявление, а не фильтр. Репитер говорит сети: "я физически нахожусь в этом регионе". Это не влияет на то, какие пакеты репитер принимает или отбрасывает — для этого есть `allowf`/`denyf`.

## Зачем нужно

### 1. Маршрутизация в сети

Когда репитер объявляет `region home ru-kda-krd`, он начинает анонсировать этот факт соседям. Соседние репитеры запоминают: "до региона ru-kda-krd можно добраться через этот репитер".

Так по цепочке анонсов строится таблица маршрутизации — каждый репитер знает, куда переслать пакет, чтобы он достиг нужного региона.

### 2. Транспортные коды в исходящих пакетах

Репитер помечает свои собственные исходящие пакеты (сервисные сообщения, keepalive, telemetry) кодом home-региона. Это позволяет другим репитерам правильно фильтровать и маршрутизировать даже служебный трафик.

### 3. Построение таблиц маршрутизации у соседей

Каждый репитер периодически рассылает анонсы своего home-региона. Соседи обновляют свои таблицы:
- Репитер A (home: ru-kda-krd) анонсирует "ru-kda-krd reachable через меня"
- Репитер B (сосед A) запоминает и сам анонсирует "ru-kda-krd reachable через A"
- Репитер C (сосед B, но не сосед A) узнаёт "ru-kda-krd reachable через B"

Так информация о достижимости распространяется по всей сети.

## Что будет без region home

Репитер продолжит пересылать пакеты по правилам `allowf`/`denyf`. Но:

- Сеть не будет знать оптимальный маршрут до его региона
- Пакеты могут идти окольными путями или теряться на дальних хоперах
- Другие репитеры не смогут корректно анонсировать достижимость этого узла
- Фильтрация на удалённых репитерах может работать некорректно — они просто не будут знать, что этот регион существует в сети

## Примеры из сети Юга России

### Карта репитеров

```
┌─────────────────────────────────────────────────────┐
│                  РОССИЯ (ru)                         │
│                                                      │
│  ┌─── Краснодарский край (ru-kda) ──────────────┐   │
│  │                                                │   │
│  │  [R1] Краснодар        home: ru-kda-krd       │   │
│  │       координаты: 45.04, 38.97                │   │
│  │                                                │   │
│  │       60 км радиоволной                       │   │
│  │                                                │   │
│  │  [R2] Горячий Ключ    home: ru-kda-gk         │   │
│  │       координаты: 44.63, 39.13                │   │
│  │                                                │   │
│  └────────────────────────────────────────────────┘   │
│                                                      │
│          ~300 км (нет прямой радиосвязи)              │
│                                                      │
│  ┌─── Ставропольский край (ru-sta) ──────────────┐   │
│  │                                                │   │
│  │  [R3] Ставрополь      home: ru-sta-stv         │   │
│  │       координаты: 45.04, 41.97                │   │
│  │                                                │   │
│  │       130 км радиоволной                      │   │
│  │                                                │   │
│  │  [R4] КМВ             home: ru-sta-kmv         │   │
│  │       координаты: 44.20, 43.08                │   │
│  │                                                │   │
│  └────────────────────────────────────────────────┘   │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### Сценарий 1: Пакет из Краснодара в Ставрополь

Пользователь в Краснодаре отправляет сообщение в канал со scope `ru`.

```
[R1] Краснодар (home: ru-kda-krd)
  │  Пакет со scope=ru
  │  R1 знает: allowf ru → разрешено
  │  R1 пересылает соседям
  ▼
[R2] Горячий Ключ (home: ru-kda-gk)
  │  Пакет со scope=ru
  │  R2 знает: allowf ru → разрешено
  │  R2 видит в таблице: ru-sta-stv reachable через R3 (или через промежуточные узлы)
  │  R2 пересылает дальше
  ▼
  ... промежуточные репитеры / шлюзы ...
  ▼
[R3] Ставрополь (home: ru-sta-stv)
  │  Пакет со scope=ru
  │  R3 знает: allowf ru → разрешено
  │  Доставлен
```

Ключевой момент: R1 знает про R3 не напрямую. Он знает что "ru-sta-stv reachable" потому что анонсы `region home` распространились по сети. Без `region home` на R3 — R1 бы не знал что Ставрополь вообще существует в сети.

### Сценарий 2: Сообщение в городской чат

Пользователь в Горячем Ключе отправляет в городской чат со scope `ru-kda-gk`.

```
[R2] Горячий Ключ (home: ru-kda-gk)
  │  Пакет со scope=ru-kda-gk
  │  R2 пересылает соседям
  ▼
[R1] Краснодар (home: ru-kda-krd)
  │  Пакет со scope=ru-kda-gk
  │  R1: allowf ru-kda-gk → разрешено
  │  R1 видит в таблице: home=ru-kda-gk это R2
  │  R1 знает что R2 уже получил этот пакет
  │  R1 НЕ пересылает дальше в сторону Ставрополя
  ▼
  Пакет остался в крае. Флуд не пошёл за пределы.
```

Здесь `region home` помогает R1 понять что пакет уже дошёл до адресата и не нужно гонять его дальше по сети.

### Сценарий 3: Между Горячим Ключом и КМВ

Пользователь в Горячем Ключе пишет в канал со scope `ru` (общероссийский).

```
[R2] Горячий Ключ (home: ru-kda-gk)
  │  Пакет со scope=ru
  ▼
[R1] Краснодар (home: ru-kda-krd)
  │  allowf ru → разрешено
  ▼
  ... маршрут через сеть ...
  ▼
[R3] Ставрополь (home: ru-sta-stv)
  │  allowf ru → разрешено
  ▼
[R4] КМВ (home: ru-sta-kmv)
  │  allowf ru → разрешено
  │  Доставлен
```

А если бы scope был `ru-kda-gk` — пакет до КМВ бы не дошёл:
```
[R2] Горячий Ключ
  │  Пакет со scope=ru-kda-gk
  ▼
[R1] Краснодар
  │  allowf ru-kda-gk → разрешено
  │  Но дальше по цепочке репитеры проверят: allowf ru-kda-gk?
  │  R3 (Ставрополь): ru-kda-gk НЕ в allowf → отброшен
  │  R4 (КМВ):        ru-kda-gk НЕ в allowf → отброшен
  ▼
  Пакет остался в Краснодарском крае. КМВ его не увидел.
```

### Сценарий 4: Пограничный репитер

Допустим, репитер на возвышенности между краями видит и краснодарские, и ставропольские узлы.

```
[R5] Пограничный (home: ru-kda-krd)
  │  allowf: ru, ru-kda, ru-sta, ru-kda-krd, ru-sta-kmv
  │
  ├── видит R1 (Краснодар)      → knows ru-kda-krd reachable
  ├── видит R3 (Ставрополь)     → knows ru-sta-stv reachable
  │
  │  home = ru-kda-krd, но allowf включает оба края
  │  Работает как мост между краями
```

Здесь `region home` определяет принадлежность узла (он "в Краснодарском крае"), а `allowf` определяет что он пересылает (оба края).

## Итог

| Команда | Что делает | Влияние на пакеты |
|---|---|---|
| `region home X` | Объявляет "я в регионе X" | Не фильтрует. Влияет на маршрутизацию и анонсы |
| `region allowf X` | Разрешает пересылку пакетов со scope X | Фильтр на входящий/транзитный трафик |
| `region denyf X` | Запрещает пересылку пакетов со scope X | Фильтр на входящий/транзитный трафик |
| `region put X` | Создаёт регион X в памяти | Нет влияния на трафик, только регистрация |
| `region default X` | Scope для DM по умолчанию | Помечает исходящие DM пакеты |
