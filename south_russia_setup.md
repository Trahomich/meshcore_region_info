# Настройка MeshCore репитеров — Юг России

## Карта регионов

```
* F                                  (корень, разрешает всё по умолчанию)
  ru F                               (Россия — суперрегион)
    ru-kda F                         (Краснодарский край)
      ru-kda-krd F                   (Краснодар)
      ru-kda-gk F                    (Горячий Ключ)
    ru-sta F                         (Ставропольский край)
      ru-sta-stv F                   (Ставрополь)
      ru-sta-kmv F                   (КМВ)
```

## Именование

| Код   | Регион / Город          |
|-------|-------------------------|
| `ru`  | Россия (суперрегион)    |
| `ru-kda` | Краснодарский край   |
| `ru-kda-krd` | Краснодар        |
| `ru-kda-gk`  | Горячий Ключ     |
| `ru-sta` | Ставропольский край  |
| `ru-sta-stv` | Ставрополь       |
| `ru-sta-kmv` | КМВ             |

> Правило: коды краёв — трёхбуквенные (kda, sta), коды городов — трёхбуквенные (krd, gk, stv, kmv).

---

## Настройка репитеров

### Краснодар

```
region put ru
region put ru-kda ru
region put ru-kda-krd ru-kda
region home ru-kda-krd
region default ru-kda-krd
region save
```

### Горячий Ключ

```
region put ru
region put ru-kda ru
region put ru-kda-gk ru-kda
region home ru-kda-gk
region default ru-kda-gk
region save
```

### Ставрополь

```
region put ru
region put ru-sta ru
region put ru-sta-stv ru-sta
region home ru-sta-stv
region default ru-sta-stv
region save
```

### КМВ (Пятигорск / Ессентуки / Кисловодск)

```
region put ru
region put ru-sta ru
region put ru-sta-kmv ru-sta
region home ru-sta-kmv
region default ru-sta-kmv
region save
```

### Пограничный репитер (охватывает два края)

Если репитер физически стоит на границе и может обслуживать оба края —
добавь подрегионы обоих:

```
region put ru
region put ru-kda ru
region put ru-sta ru
region put ru-kda-krd ru-kda
region put ru-sta-kmv ru-sta
region home ru-kda-krd
region default ru-kda-krd
region save
```

---

## Фильтрация в клиентском приложении

1. Открой канал → три точки → **Set Region Scope**
2. Добавь нужные регионы из списка выше
3. Выбери scope для канала:

| Канал                 | Scope       |
|-----------------------|-------------|
| Общероссийский чат    | `ru`        |
| Чат Краснодарского края | `ru-kda`  |
| Чат Ставропольского края | `ru-sta` |
| Чат Краснодара        | `ru-kda-krd`|
| Чат Горячего Ключа    | `ru-kda-gk` |
| Чат Ставрополя        | `ru-sta-stv`|
| Чат КМВ               | `ru-sta-kmv`|

4. Приложение запомнит scope для каждого канала отдельно.

---

## Важные замечания

- Начиная с v1.15.0 команда `region put` автоматически выполняет `allowf` — отдельная команда не нужна.
- Имена регионов: только строчные латинские буквы, цифры и дефис, до 29 байт.
- DM-сообщения **не фильтруются** регионами — регионы работают только для каналов.
- Шаблон `* F` (по умолчанию) разрешает пересылку всех пакетов. Для жёсткой фильтрации: `region denyf *`, но только после того как все отправляют сообщения со scope.
- `region default` задаёт scope для исходящих сообщений без явного указания (v1.15.0+).

---

## Порядок развёртывания

1. Настроить все репитеры (`region put`, `home`, `default`, `save`)
2. Убедиться что репитеры видят друг друга
3. Попросить пользователей выставить scope в каналах
4. После проверки — при необходимости закрыть `*` через `region denyf *`

## ЧАВО

### Что означает `region home ru-kda-krd`?

`region home` — это declaring (объявление) местоположения репитера в иерархии регионов. Это **не фильтр** и **не правило пересылки**. Это метка, которая говорит: "данный репитер физически находится в регионе ru-kda-krd (Краснодар)".

**Зачем это нужно:**

1. **Маршрутизация пакетов.** Когда узел сети хочет отправить сообщение в регион ru-kda-krd, mesh-сеть должна знать, через какой репитер этот регион reachable. Если репитер в Краснодаре объявил `region home ru-kda-krd`, сеть понимает, что пакеты со scope `ru-kda-krd` нужно направлять через этот репитер.

2. **Транспортные коды отправителя.** Когда пользователь, подключённый к этому репитеру, отправляет сообщение в канал без явного указания scope, используется `region default`. А `region home` определяет, какие транспортные коды подставляются в исходящие пакеты от самого репитера (если он выступает как клиентский узел).

3. **Информирование соседей.** Репитер регулярно анонсирует свой home-регион соседним узлам. Это позволяет другим репитерам строить таблицы маршрутизации — они знают, что для доставки в ru-kda-krd нужно пересылать пакеты через репитер Краснодара.

**Что будет, если не задать `region home`:**

- Репитер всё равно будет пересылать пакеты в соответствии со своими `allowf` правилами
- Но сеть не будет знать, через какой репитер reachable конкретный регион
- Пакеты со scope города могут приходить с задержкой или не приходить вообще — сеть не знает оптимального маршрута
- Команда `region default` не сможет корректно работать — она опирается на `region home`

**Пример:**

Репитер в Краснодаре:
```
region home ru-kda-krd
```

Репитер в Горячем Ключе:
```
region home ru-kda-gk
```

Когда кто-то в Ставрополе отправляет сообщение со scope `ru-kda-krd`, сеть знает: пересылать через репитер Краснодара (он объявил `home ru-kda-krd`). Репитер Горячего Ключа это сообщение не получит — оно не для его региона.

### Что означает `*` (шаблонный регион) — это "все scope" или "без scope"?

**`*` — это "без scope"** (точнее — fallback для пакетов без транспортных кодов).

`*` — это шаблонный регион (wildcard), корень всей иерархии. Он работает как "регион по умолчанию" и обрабатывает пакеты, у которых **нет явного транспортного кода**.

**Логика пересылки на репитере:**

1. Пакет пришёл **с транспортным кодом** (отправитель указал scope, например `ru-kda-krd`):
   - Репитер ищет регион, чей хеш совпадает с транспортным кодом
   - Нашёл → проверяет `allowf` для этого региона → пересылает или отбрасывает
   - Не нашёл → fallback на `*` → проверяет `allowf` для `*` → пересылает или отбрасывает

2. Пакет пришёл **без транспортного кода** (отправитель не указал scope):
   - Сразу fallback на `*`
   - Если `* F` (allowf включён) → пересылает
   - Если `*` без `F` (denyf) → отбрасывает

**Что это значит на практике:**

| Состояние `*` | Поведение |
|---|---|
| `* F` (по умолчанию) | Все пакеты без scope пересылаются + пакеты с неизвестным scope тоже проходят. Сеть полностью открыта. |
| `*` без `F` (`denyf *`) | Пакеты без scope **отбрасываются**. Пакеты с неизвестным scope тоже отбрасываются. Пройти могут только пакеты с известным scope и `allowf`. |

**Почему `*` НЕ означает "все scope":**

`*` не объединяет все регионы. Он обрабатывает только два случая:
- Пакеты, отправленные вообще без scope
- Пакеты со scope, который репитер не знает (нет такого региона в его карте)

Если пакет отправлен со scope `ru-kda-krd`, он будет обработан регионом `ru-kda-krd`, а не `*`. `*` в это время "не участвует".

**Рекомендация для нашей сети:**

На первом этапе оставляем `* F` — всё работает как обычно, пакеты без scope проходят везде. Когда все пользователи начнут указывать scope в каналах — закрываем шаблон: `region denyf *`. После этого пакеты без scope перестанут пересылаться, и фильтрация заработает в полную силу.

### Как будут ходить пакеты между Ставрополем и Краснодаром? Между Горячим Ключом и КМВ?

Зависит от scope, с которым отправлено сообщение:

**Scope `ru` (общероссийский)** — пакет пройдёт через все репитеры, у которых есть `allowf ru`. А он есть у всех по нашей схеме. Сообщение доставляется из Краснодара в Ставрополь и обратно без ограничений.

**Scope `ru-kda` (Краснодарский край)** — пакет пересылают только репитеры с `allowf ru-kda`, то есть репитеры в Краснодарском крае (Краснодар, Горячий Ключ). В Ставропольский край пакет не попадёт.

**Scope `ru-kda-krd` (Краснодар)** — пакет пересылают только репитеры с `allowf ru-kda-krd`. Только репитер в Краснодаре.

**Конкретные примеры:**

| Отправитель | Получатель | Scope | Доставлено? |
|---|---|---|---|
| Краснодар → Ставрополь | общий чат | `ru` | Да, через все репитеры |
| Краснодар → Ставрополь | чат края | `ru-kda` | Нет — репитеры Ставропольского края не пересылают `ru-kda` |
| Горячий Ключ → КМВ | общий чат | `ru` | Да |
| Горячий Ключ → КМВ | чат края | `ru-kda` | Нет — КМВ в `ru-sta` |
| Горячий Ключ → Краснодар | чат города | `ru-kda-krd` | Нет — Горячий Ключ только `ru-kda-gk`, не пересылает `ru-kda-krd` |
| Горячий Ключ → Краснодар | чат края | `ru-kda` | Да — оба репитера в `ru-kda` |

**Вывод:** для межкраевой связи используй scope `ru`. Для внутрикраевой — scope края. Для внутригородской — scope города.

### Насколько уменьшится флуд в сети?

Зависит от того, насколько активно используется фильтрация. При текущей конфигурации (с `* F` по умолчанию) флуд не уменьшится — шаблон `*` разрешает пересылку всего.

Фильтрация начнёт работать, когда:

1. Пользователи выставят scope в каналах
2. Шаблон `*` будет закрыт командой `region denyf *`

**Примерная оценка после включения фильтрации:**

Предположим, 6 репитеров, трафик распределён равномерно:

| Тип сообщения | Без фильтрации | С фильтрацией | Снижение |
|---|---|---|---|
| Городской чат | 6 репитеров | 1 репитер | ~83% |
| Чат края (3 репитера в крае) | 6 репитеров | 3 репитера | ~50% |
| Общероссийский чат | 6 репитеров | 6 репитеров | 0% |
| DM (личные сообщения) | 6 репитеров | 6 репитеров | 0% |

**Реальный эффект:** при типичном использовании, где ~70% трафика — городские чаты, ~20% краевые, ~10% общие, общее снижение пересылаемого трафика составит примерно **60-70%**.

**Важно:** DM-сообщения не фильтруются регионами — они всегда расходятся по всей сети.
